# PROJECT_ROADMAP.md 重构设计

## 目标

将现有 `PROJECT_ROADMAP.md` 从“历史修复记录 + 当前任务 + 长期方向”混杂的滚动文档，重构为一份适合新一轮优化使用的执行型路线图。

本轮路线图的主轴明确为：

- 提取鲁棒性优先；
- 完整归档上一轮已完成成果；
- 对下一轮给出可直接执行的阶段计划；
- 仅在附录中保留中长期方向，避免稀释当前优先级。

## 背景

当前路线图已经较完整地记录了上一轮优化成果，包括：

- 提取规则修复与字段覆盖补强；
- 页码证据链与“待核验”机制；
- 风险评分配置化与输出一致性；
- CLI、打包、日志与测试回归收口。

这些内容对理解项目演进有价值，但继续沿用原来的滚动勾选方式会带来两个问题：

1. 历史成果与当前优先级混在一起，读者难以判断当前主战场；
2. 新一轮若继续在原结构上追加任务，文件会再次膨胀并失去维护边界。

因此需要一次结构性重写，而不是继续在原文上增补条目。

## 范围

### 本次重构包含

- 重写 `PROJECT_ROADMAP.md` 的整体结构；
- 将上一轮已完成优化整理为“历史基线”归档区；
- 将“提取鲁棒性优先”定义为当前执行主线；
- 为下一轮写出分阶段执行计划；
- 增加路线图维护规则，防止后续再次失控；
- 保留中长期方向，但降级为附录，不进入当前执行清单。

### 本次重构不包含

- 修改项目源代码；
- 立即展开新的实现计划或任务分解文档；
- 删除历史优化成果，只会调整其组织方式；
- 将分析能力深化或产品化事项提升为当前轮的主执行目标。

## 设计原则

### 1. 历史与当前分离

路线图必须先回答“项目已经解决了什么”，再回答“当前正在解决什么”。因此上一轮成果统一进入归档区，不再和当前待办混排。

### 2. 当前主线唯一

正文只服务一个主目标：提取鲁棒性优化。分析能力深化与产品化只能作为后续方向存在，不能在当前阶段与主线抢优先级。

### 3. 执行导向

下一轮路线图使用固定的执行结构，每一阶段都包含：

- 目标；
- 关键任务；
- 完成标准；
- 风险与依赖。

这样后续实现时可以直接把路线图转化为计划，而不必先再次解释意图。

### 4. 维护边界清晰

路线图必须定义更新规则，约束后续只在“当前执行路线图”中移动、勾选和归档内容，避免再次回到无边界追加状态。

## 新文档结构

## 1. 文档用途

简要说明这是项目的长期维护路线图，并明确：

- 当前执行轮次是什么；
- 什么情况下需要更新路线图；
- 路线图不等于实现计划，而是执行优先级与阶段边界的定义文件。

## 2. 历史基线：上一轮已完成优化（归档）

这一部分完整保留上一轮成果，但改为按能力域归类，而不是一长串流水项。建议分为以下小节：

### 提取准确性修复

归档字段提取修复、版式特例支持、误提取抑制等成果。

### 证据链与待核验机制

归档页码证据、附注证据、报告信号分层、“待核验”投资建议降级等成果。

### 配置化与评分一致性

归档阈值/权重接线、风险评分一致性修复、规则配置化等成果。

### CLI、打包与日志收口

归档共享 CLI、输出格式修复、打包一致性、日志替换等成果。

### 测试与回归资产

归档样本夹具、回归测试、烟雾测试和元数据一致性测试等成果。

这一部分的写法应当体现“这些是当前系统已经具备的底座”，而不是继续作为待办清单使用。

## 3. 当前轮次：提取鲁棒性优先

这是路线图正文核心，明确说明本轮目标是提升在真实 PDF 年报场景下的稳定性、覆盖率和证据完整性。

该部分应先写出：

- 当前问题定义；
- 本轮优化目标；
- 本轮不优先处理的事项。

### 当前问题定义

重点指出：

- PDF 提取仍高度依赖正则，抗版式扰动能力不足；
- 部分关键字段和附注证据仍存在覆盖缺口；
- 样本回归虽已建立，但对“版式族群”覆盖还不够；
- 并行批处理和产品化接口不是当前轮主目标。

### 本轮优化目标

- 降低版式变化对提取结果的破坏；
- 提升核心字段和附注字段的覆盖率；
- 让证据缺口可见、可测、可回归；
- 为后续分析能力深化提供更稳定的数据底座。

## 4. 下一轮执行路线图

该部分写成三个执行阶段。

### 阶段 A：解析稳定性与版式抗扰动

目标：
提高主表和附注在不同年报版式下的解析稳定性。

关键任务：

- 识别当前规则中最脆弱的版式假设；
- 将高频字段提取从单一路径规则升级为多策略匹配；
- 加强对目录页、分析区、表头噪声和跨行文本的隔离；
- 为新三板、小体量企业和非标准版式建立独立兜底策略。

完成标准：

- 常见已知版式下核心字段提取稳定；
- 新增的兜底逻辑被测试覆盖；
- 已知误提取场景在回归集中不再复发。

风险与依赖：

- 需要真实样本支撑，不适合仅靠合成文本调规则；
- 规则增多可能引入相互干扰，需要阶段性回归。

### 阶段 B：字段覆盖与证据完整性

目标：
补齐当前分析链路依赖但仍易缺失的字段与证据引用。

关键任务：

- 梳理所有当前规则依赖字段，标记高价值缺口；
- 优先补足流动性、现金流、治理类、附注类关键字段；
- 强化字段值与页码/摘录证据的一一对应；
- 明确“提取到值但无证据”与“未提取到值”的不同状态。

完成标准：

- 当前高价值风险规则依赖字段具备可接受覆盖率；
- 报告中的关键风险信号优先引用主表或附注原文；
- 缺失数据在输出层可见，不再被静默吞掉。

风险与依赖：

- 个别字段可能天然难以稳定抽取，需要允许保守降级；
- 证据完整性要求会推高实现复杂度，需和误报率平衡。

### 阶段 C：验证体系与回归资产扩充

目标：
把“提取鲁棒性”从规则感觉转成可验证、可回归的质量体系。

关键任务：

- 按版式族群整理样本，而不是只按公司堆积样本；
- 扩大 PDF 回归夹具，覆盖更多主表/附注变体；
- 增加字段覆盖率、证据覆盖率、误提取抑制类断言；
- 建立新增样本进入回归集的准入规则。

完成标准：

- 新增样本能反映主要版式族群；
- 新规则上线前可通过回归集验证；
- 回归测试不只校验数值，还校验证据与缺失状态。

风险与依赖：

- 样本维护成本会持续上升；
- 需要控制测试运行时间，避免回归集膨胀后失去可用性。

## 5. 后续阶段（非本轮）

这一部分只保留方向，不写成本轮执行项。

### 分析能力深化

- 多年趋势提取与趋势分析；
- 行业基准与同行对比；
- 跨报表一致性规则引擎；
- 提取置信度与结论置信度表达。

### 产品化推进

- 真正的批量并行处理；
- 稳定的结构化 API/服务层；
- 面向下游系统的输出协议；
- 更规范的环境配置与部署方式。

## 6. 维护规则

路线图结尾应增加维护约束：

1. 当前执行轮只允许在“下一轮执行路线图”部分更新状态和优先级；
2. 当整轮完成后，再将本轮成果整体迁入“历史基线”；
3. “后续阶段（非本轮）”中的事项只有在当前轮收尾后才可升级为正文执行阶段；
4. 若出现新的高优先级问题，必须先判断它是否属于当前主轴，避免再次混入跨主题任务。

## 成功标准

重构后的 `PROJECT_ROADMAP.md` 应满足以下标准：

- 读者能快速区分“已经完成的底座”和“当前正在推进的主线”；
- 当前路线图正文只聚焦提取鲁棒性；
- 每个执行阶段都具备明确的目标、任务、完成标准和风险；
- 中长期方向被保留，但不会干扰当前执行；
- 文档具备可持续维护的更新规则。
