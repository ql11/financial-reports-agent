# 项目路线图

## 文档用途

这是 `financial-reports-agent` 的长期维护路线图。

本文件只负责定义三件事：

- 项目已经具备的能力基线；
- 当前正在推进的主优化轮次；
- 当前轮次之后的中长期方向。

出现以下情况时，需要同步更新本文件：

- 当前轮次整体完成，需要迁入历史基线；
- 当前轮次内部的阶段顺序、目标或优先级发生变化；
- 识别出新的高优先级问题，并确认它属于当前主轴；
- 当前轮次收尾，准备将下一阶段主题升级为新的执行轮次。

本文件不是实现计划，也不是任务工单列表。它用于定义执行边界、优先级和阶段目标，后续实现计划应基于本文件单独展开。

## 历史基线：上一轮已完成优化（归档）

上一轮优化的核心目标是先把输出结果做得可解释、可回溯、可验证，并修补明显影响可信度的流程缺陷。以下内容已经完成，构成当前系统的已知能力底座。

### 提取准确性修复

- 去掉了分析器内部的隐式单文件报告保存，避免单文件分析重复落盘。
- 修复了批量分析不尊重 `--format` 参数的问题，单文件与批处理的输出契约已一致。
- 补全了已经被下游规则使用但此前未计算的周转率字段。
- 补充了 `total_assets`、`total_liabilities`、`total_equity`、`accounts_receivable`、`net_cash_flow_financing` 等关键字段的提取规则。
- 针对真实样本补充了股票代码、流动性字段、现金流字段等关键字段的提取覆盖。
- 修复了小金额净利润被错误阈值过滤的问题，降低了对新三板和小体量企业的误伤。
- 针对“归属于挂牌公司股东的净利润/净资产”等常见表述增加了专门提取规则。
- 增强了主表提取时“优先匹配报表正文、避开经营分析区和比例列污染”的策略。
- 补足了“仅有当年值”“附注编号夹在字段名与金额之间”“单行主表记录”等场景的提取规则。
- 收紧了附注提取范围，修复了政府补助、坏账准备、存货跌价准备等串段误提取问题。
- 过滤了目录页样式和“关联交易 100%”类噪声，避免误触发治理类字段。

### 证据链与待核验机制

- 报告已经支持在风险信号下展示页码证据和原文摘录。
- 审计意见原文已接入页码证据链，修复了“带持续经营重大不确定性段落的无保留意见”被误判为标准无保留意见的问题。
- 主表关键字段已经补齐页码证据锚点，报告可优先引用主表原文而不是只依赖附注。
- 坏账准备、存货跌价准备等已提取附注数据已进入保守型规则，不再只是静态展示。
- 在证据链不完整时，自动投资建议会降级为“待核验”，避免直接输出低风险增持结论。
- “待核验”场景下的建议口径已经收口，避免自动建议与保守提示同时出现而互相冲突。
- 会计政策变更、会计估计变更、历史违规、审计师变更、或有事项、表外融资、研发资本化、勾稽异常、现金存量与利息错配等治理类/附注类信号已经接入 `note:*` 页码证据链。
- 输出层已区分“明确异常”“弱信号”“缺失数据”，分析结论的保守性和可解释性显著提升。

### 配置化与评分一致性

- 主要异常检测阈值和风险评分权重已经接入 `configs/thresholds.yaml` 与 `configs/weights.yaml`。
- `FraudDetector` 中高频规则和治理类规则已完成运行时配置化，覆盖业绩背离、现金流背离、应收/存货异常、政府补助、关联交易、产能异常、资产高估、报表勾稽、现金流质量、会计政策/估计变更、审计意见、审计师变更和历史违规等主题。
- 修复了运行时风险评分与序列化输出评分不一致的问题。
- 风险评分公式、分项得分和最大风险加分逻辑已经稳定，报告能展示更透明的评分拆解。

### CLI、打包与日志收口

- `requirements.txt` 已收敛为运行时依赖。
- `pyproject.toml` 与 `setup.py` 的入口配置已经统一到共享 CLI 模块。
- README 已补充 `pip install -e .` 安装方式、共享 console scripts 用法和 CLI 安装验证命令。
- 核心分析链路已统一改为共享 `logging` 输出，`analyzer`、`data_extractor`、`report_generator` 不再直接使用 `print`。

### 测试与回归资产

- 新增并收敛了覆盖上述修复点的回归测试。
- 建立了真实 PDF 样本夹具，覆盖主表、附注证据和共享 CLI 入口的回归场景。
- 增加了 CLI 烟雾测试、项目元数据一致性测试和日志行为测试。
- 全量测试已能验证提取逻辑、欺诈检测规则、评分输出、报告格式和安装入口的一致性。

## 当前轮次：提取鲁棒性优先

### 当前问题定义

经过上一轮优化，项目已经从“可运行”提升到“具备基本可信度”。但当前最主要的瓶颈仍然不是规则数量，而是提取层在真实 PDF 年报场景下的鲁棒性。

当前主要问题包括：

- PDF 提取仍高度依赖正则和局部文本模式，对版式变化、跨行断裂、表头扰动和章节噪声较敏感。
- 一些高价值字段虽然已有规则，但在不同报告模板下仍可能缺失、误提取或缺少证据锚点。
- 回归测试已经建立，但对“版式族群”的覆盖仍不够，更多是按个别样本修补问题。
- 分析能力深化和产品化虽然重要，但如果提取底座不够稳定，这些能力会建立在脆弱输入之上。

### 本轮优化目标

本轮只聚焦一个主方向：提升提取鲁棒性。

具体目标如下：

- 降低版式变化对主表和附注提取结果的破坏；
- 提升当前分析链路依赖字段的覆盖率与稳定性；
- 提高字段值与页码/摘录证据之间的对应完整性；
- 将“缺失、误提取、弱证据”转化为可见、可测、可回归的问题；
- 为下一轮分析能力深化提供更稳定的真实数据底座。

### 本轮不优先处理的事项

以下事项保留为后续方向，但不进入当前轮次的执行主线：

- 多年趋势分析与同行对比；
- 大规模跨报表一致性规则扩展；
- 真正的批量并行处理；
- 稳定的 API/服务层与下游结构化协议；
- 完整的产品化部署与环境体系。

## 下一轮执行路线图

### 阶段 A：解析稳定性与版式抗扰动

#### 目标

提高主表和附注在不同年报版式下的解析稳定性，优先解决“规则对单一样本有效，但一换模板就失效”的问题。

#### 关键任务

- 识别当前主表、附注、治理信息提取中最脆弱的版式假设。
- 将高频字段提取从单一路径规则升级为多策略匹配，而不是继续堆叠单条正则。
- 强化对目录页、经营分析区、表头噪声、比例列、跨行文本和注释编号干扰的隔离。
- 为新三板、小体量企业、非标准主表标题和简化披露模板建立独立兜底策略。
- 梳理“正文提取优先”和“附注提取优先”的适用边界，避免二者相互污染。

#### 完成标准

- 常见已知版式下的核心字段提取结果稳定，不再依赖样本级热修复。
- 已知误提取场景能够被新策略覆盖，并在回归集中不再复发。
- 新增的兜底逻辑具备明确测试，不是只在代码中临时插入分支。
- 同一字段在不同模板下的提取路径更一致，降低后续维护成本。

#### 风险与依赖

- 这一阶段高度依赖真实样本，仅靠合成文本无法充分验证版式问题。
- 多策略匹配会增加逻辑分叉，若没有足够回归约束，容易引入相互干扰。
- 需要克制对单一样本的过拟合，避免“修一个坏一个”。

### 阶段 B：字段覆盖与证据完整性

#### 目标

补齐当前分析链路依赖但仍容易缺失的关键字段，并强化字段与证据锚点之间的一一对应关系。

#### 关键任务

- 梳理所有现有欺诈规则、异常检测和报告逻辑依赖的输入字段，标记高价值缺口。
- 优先补足流动性、现金流、治理类、附注类和交叉验证类字段。
- 将“提取到值但无证据”“提取失败”“证据存在但值不稳定”区分成不同状态。
- 强化主表字段与页码证据、附注字段与上下文摘录之间的对应完整性。
- 对仍难稳定提取的字段建立保守降级策略，避免静默失败后继续参与高置信度结论。

#### 完成标准

- 当前高价值风险规则依赖字段具备可接受覆盖率，明显缺口被显式记录而非隐藏。
- 报告中的关键风险信号能优先回溯到主表或附注原文。
- 缺失数据和证据缺口能稳定出现在输出层，用户能明确看到结论边界。
- “待核验”触发条件更精确，既不过度泛化，也不会漏掉高风险证据缺口。

#### 风险与依赖

- 某些字段在原始 PDF 中天然难以稳定提取，必须接受保守处理而非强行给值。
- 证据完整性要求会推高实现复杂度，需要与误报率、维护成本做平衡。
- 若字段清单梳理不完整，后续很容易重复补洞。

### 阶段 C：验证体系与回归资产扩充

#### 目标

把“提取鲁棒性”从经验判断转成一套可以持续验证、持续回归的质量体系。

#### 关键任务

- 按版式族群整理样本，而不是只按公司名堆积 PDF。
- 扩大 PDF 回归夹具，覆盖更多主表结构、附注布局、治理披露模板和噪声场景。
- 增加字段覆盖率、证据覆盖率、误提取抑制类断言，而不只校验最终数值。
- 建立新增样本纳入回归集的准入规则和标注规范。
- 控制回归集规模与运行时间，避免验证体系膨胀后难以使用。

#### 完成标准

- 新增样本能覆盖主要版式族群，而不是继续围绕少数典型样本修补。
- 新规则上线前可以通过回归集验证主要风险场景。
- 回归测试同时覆盖数值、证据和缺失状态。
- 样本维护方式清晰，后续新增回归案例有统一入口。

#### 风险与依赖

- 样本扩充会持续增加维护成本，需要建立最小必要集而非无限扩张。
- 如果测试运行时间失控，回归体系会被绕开，最终失去约束力。
- 样本标签标准若不统一，会削弱断言可信度。

## 后续阶段（非本轮）

### 分析能力深化

当前轮次完成后，优先考虑将以下主题升级为下一轮执行主线：

- 多年趋势提取与趋势分析；
- 行业基准与同行对比；
- 跨报表一致性规则引擎；
- 提取置信度与结论置信度表达；
- 更细粒度的信号解释与因子拆解。

### 产品化推进

在提取鲁棒性和分析底座稳定后，再推进以下事项：

- 真正的批量并行处理与进度反馈；
- 稳定的结构化 API/服务层；
- 面向下游系统的输出协议；
- 更规范的环境配置、运行模式与部署方式；
- 更成熟的报告模板与导出形态。

## 维护规则

为避免路线图再次回到“历史、当前、未来混杂追加”的状态，后续维护遵循以下规则：

1. 当前轮次只允许在“下一轮执行路线图”部分调整状态、顺序和优先级。
2. 当整轮完成后，再将本轮成果整体迁入“历史基线”，而不是直接在正文里无限打勾追加。
3. “后续阶段（非本轮）”中的事项只有在当前轮次收尾后，才可升级为新的执行轮次。
4. 若出现新的高优先级问题，必须先判断它是否属于“提取鲁棒性优先”这一主轴；不属于则记录到后续阶段，不直接打断当前主线。
5. 新的实现计划、任务拆解和执行跟踪，应单独写入计划文档，不应直接塞回本路线图正文。
